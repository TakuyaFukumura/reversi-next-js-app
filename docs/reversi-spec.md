# リバーシ実装検討資料

## 1. 目的

- Next.js アプリ上で動作するシンプルなリバーシ（オセロ）ゲーム画面を実装するため、実装前に仕様を整理する
- 本資料では UI、状態管理、ゲーム進行、NPC 思考ロジック、コンポーネント分割の方針を定義する
- 本 issue では実装は行わず、仕様検討のみに専念する

## 2. スコープ

### 対象

- 8x8 固定ボードのリバーシゲーム画面
- 人間 vs NPC の 1 人プレイ
- フロントエンドのみで完結する状態管理
- スマホ対応を含むレスポンシブ UI

### 対象外

- 人間 vs 人間
- オンライン対戦
- データベース保存
- 対局履歴の永続化
- 複数難易度

## 3. 画面仕様

### 3.1 画面構成

- タイトル
- 現在の手番表示
- 石数表示（人間 / NPC）
- 勝敗またはゲーム終了メッセージ
- 8x8 の盤面
- 操作ボタン
  - パス
  - 終了
  - リセット

### 3.2 表示ルール

- 合法手があるセルのみクリック可能
- 合法手の候補は視認できる形で表示する
- NPC 思考中は盤面操作を無効化する
- ゲーム終了時は勝者または引き分けを表示する

### 3.3 レスポンシブ方針

- スマホでは縦積みレイアウトを基本とする
- 盤面は画面幅に応じて正方形を維持する
- ボタンはタップしやすい高さと余白を確保する
- タブレット以上では情報パネルと盤面の余白を広めに取る

## 4. ゲーム仕様

### 4.1 初期配置

- ボードサイズは 8x8
- 初期配置は標準リバーシと同様に中央 4 マスへ配置する
- 人間が先手（黒）、NPC が後手（白）を初期案とする

### 4.2 手番進行

1. 現在手番の合法手を算出する
2. 合法手がある場合は 1 手選んで着手する
3. 合法手がない場合は、手番に応じてパス処理を行う
4. 裏返し処理またはパス処理の完了後、相手手番へ切り替える
5. 両者とも連続して合法手がない場合はゲーム終了とする

### 4.3 パスと終了

- 人間手番では、合法手が存在しない場合のみ手動パスボタンを有効にする
- 合法手がある状態で人間はパスを行えない
- 人間がパスボタンを押した時点で手番を NPC に切り替える
- NPC 手番で合法手がない場合は、ボタン操作なしで自動的にパスする
- 人間のパス直後に NPC も合法手を持たない場合は、その時点で終局する
- 終了ボタンは対局を中断し、結果表示エリアへ「終了済み」を表示する
- 手動終了時は勝敗判定を行わず、石数はその時点の盤面から算出した値を参考表示する
- 手動終了後は盤面操作、パスボタン、終了ボタンを無効化し、リセットボタンのみ有効とする
- リセットボタンは初期状態に戻し、新しい対局を開始する

## 5. NPC 仕様

### 5.1 思考方式

- 1 手先読みの簡易ミニマックスを採用する
- NPC の各合法手について、次の流れで評価する
  1. NPC が候補手を打った盤面を生成する
  2. その盤面で人間の合法手を列挙する
  3. 人間が最も有利になる手を選ぶと仮定する
  4. その結果に対して NPC 視点の評価値を求める
  5. 最終的に評価値が最も高い手を採用する

### 5.2 評価関数

- 初期実装は `NPC の石数 - 人間の石数` を採用する
- 将来的には以下の拡張を可能にする前提で関数を分離する
  - 隅の優先
  - 辺の優先
  - 可動手数
  - 終盤重み付け

### 5.3 同点時の扱い

- 合法手の列挙順は、上段から下段、各行は左から右へ走査する行優先順（row-major order）で固定する
- 評価値が同じ候補手が複数ある場合は、上記の列挙順で先に見つかった手を採用する
- 将来的にランダム要素を追加できるよう、手選択処理は独立させる

## 6. 状態管理方針

- React の `useReducer` を第一候補とする
- 理由
  - ゲーム進行が「着手」「パス」「リセット」「終了」などイベント駆動で整理しやすい
  - 盤面、手番、終了状態など相互に関連する状態を一元管理しやすい

### 6.1 状態の正（ソース・オブ・トゥルース）

- 永続的に保持する状態は最小限とし、以下を正とする
  - 盤面データ（8x8 二次元配列）
  - 現在手番
  - ゲーム状態
  - 結果表示メッセージ
- 人間 / NPC の石数、現在手番の合法手一覧は盤面と手番から導出する
- 合法手一覧は「着手後」「パス後」「リセット後」など盤面または手番が変わるタイミングで再計算する
- 石数は表示更新時に盤面から算出し、派生データを別状態として二重管理しない

### 6.2 想定状態

- 盤面データ（8x8 二次元配列）
- 現在手番
- ゲーム状態
  - 対局中
  - NPC 思考中
  - 終局
  - 手動終了
- 結果表示メッセージ

### 6.3 派生データ

- 人間 / NPC の石数
- 現在手番の合法手一覧
- 盤面に表示する着手候補

### 6.4 永続化

- 状態はメモリ上のみで管理する
- ページリロード時は初期状態へ戻る
- localStorage や DB への保存は行わない

## 7. コンポーネント分割案

- `GamePage`
  - 画面全体のレイアウト
- `GameStatus`
  - 手番、石数、メッセージ表示
- `Board`
  - 盤面全体
- `BoardCell`
  - 1 マス単位の表示とクリック処理
- `GameControls`
  - パス / 終了 / リセット

### 7.1 補助ロジック

- 盤面生成
- 合法手判定
- 着手処理と反転処理
- 石数集計
- 勝敗判定
- NPC 手選択

UI コンポーネントとゲームロジックは分離し、テストしやすい構成にする

## 8. 想定フロー

1. 初期盤面を表示する
2. 人間の合法手を表示する
3. 人間が着手する
4. 盤面更新後、NPC の合法手があれば NPC 思考へ移る
5. NPC が着手したら再び人間手番へ戻る
6. どちらかが打てない場合はパス処理を行う
7. 双方が打てなくなったら終局し、石数で勝敗を判定する

## 9. テスト観点

- 初期盤面が正しく表示されること
- 合法手が正しく算出されること
- 着手時に対象方向の石が正しく反転すること
- 合法手がない場合のみパス可能であること
- 人間は手動パス、NPC は自動パスになること
- NPC が合法手の中から 1 手先読み結果に基づいて手を選ぶこと
- 同点時に行優先順（row-major order）で手が選ばれること
- 終局時に勝敗表示が正しいこと
- 手動終了時に勝敗判定を行わず、盤面操作が無効になること
- リセットで初期状態に戻ること
- スマホ幅でも盤面とボタンが崩れないこと

## 10. 実装時の補足

- まずはシンプルな UI を優先し、装飾は最小限とする
- 盤面やセルは再利用可能なコンポーネントとして切り出す
- NPC ロジックは今後の評価関数差し替えを考慮して純粋関数化する
- 現在のアプリで利用している DB 前提の仕組みは、本機能では利用しない
